product-review
A collaborative product review skill for early-stage startup PMs. Feed it a plan, spec, or feature description and it walks through every branch — surfacing gaps, edge cases, and open questions focused entirely on the product and user experience.

Inspired by grill-me, but a little less intense.

What makes it different from grill-me:

Scoped to product thinking — flows, states, naming, scope, and what's missing. Never dips into technical implementation. This isn't going to help you scope join tables.
No trick questions — assumes reasonable people following reasonable workflows. Won't invent scenarios where users act against their own obvious interests. This isn't going to ask "what if engineers do P2 elements before even reading the P0 section?"
No boundary-pushing — collaborative partner, not adversarial interrogator. Says its piece once, then respects your call. Should not ever say "I was testing a boundary, glad you kept it."
Upfront question map — before asking anything, presents a rough outline of all planned questions grouped by topic so you know what you're getting into. So that you have a base understanding of how long it's going to take.
Tracks implications as you go — flags when an answer changes a previous decision, resolves an upcoming question, or opens up brand-new territory the spec didn't cover. Should prevent you from making a decision that throws the entire spec off without catching that change.
